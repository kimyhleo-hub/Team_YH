# Performance Agent — 설계 계획

## 목적

공모전 및 연구에서 ML/DL 모델 성능을 극대화하기 위한 Claude Code 에이전트 시스템.
범용 에이전트(`Performance_Agent/CLAUDE.md`)가 공통 전략과 역할을 정의하고,
각 프로젝트 폴더의 `CLAUDE.md`가 이를 상속·확장하여 해당 태스크에 특화된 최적화를 수행한다.

## 범용 에이전트 설계 (`Performance_Agent/CLAUDE.md`)

### 역할 정의

범용 에이전트는 다음 여덟 가지 역할을 수행한다.

#### 1. 하이퍼파라미터 튜닝
- 현재 설정값을 읽고 탐색 우선순위를 판단한다.
- Grid / Random / Bayesian 중 상황에 맞는 전략을 선택한다.
- 실험 결과를 보고 다음 탐색 방향을 제안한다.
- 학습률 스케줄러, 옵티마이저 종류, 정규화 강도 등을 포함한다.

#### 2. 모델 아키텍처 개선
- 현재 아키텍처의 병목(과적합, 표현력 부족, 수렴 불안정)을 진단한다.
- 레이어 구성, Attention 구조, Skip Connection, Normalization 방식을 검토한다.
- 앙상블 전략(Stacking, Blending, Snapshot)을 평가·제안한다.
- 사전학습 모델 활용 및 Fine-tuning 전략을 포함한다.

#### 3. 데이터·피처 엔지니어링
- EDA 결과 및 데이터 분포를 분석한다.
- 결측치 처리, 이상치 제거, 정규화·표준화 방식을 검토한다.
- 도메인 지식 기반 피처 생성 및 선택 전략을 제안한다.
- 데이터 증강(Augmentation) 기법을 태스크 유형에 맞게 적용한다.

#### 4. 실험 관리 및 분석
- 실험 로그(파라미터, 지표, 시간)를 구조화된 형태로 기록한다.
- 실험 간 성능 차이의 원인을 분석한다.
- 가장 유망한 방향을 우선순위와 함께 정리한다.
- 재현성 확보를 위한 시드 고정 및 환경 기록을 확인한다.

#### 5. 손실 함수 설계
- 태스크 유형과 평가지표에 맞는 손실 함수를 선택·설계한다.
- 클래스 불균형 상황에서 Focal Loss, Weighted Loss 등을 적용한다.
- 평가지표와 손실 함수의 정렬 여부를 검토한다 (예: F1 최적화 목적의 커스텀 loss).
- Contrastive Loss, Triplet Loss 등 표현 학습 목적의 손실 함수를 고려한다.
- Label Smoothing, Mixup Loss 등 정규화 효과를 가진 손실 함수를 검토한다.

#### 6. 추론·후처리 최적화
- TTA(Test Time Augmentation)를 태스크에 맞게 설계하고 적용한다.
- 분류 문제에서 임계값(Threshold)을 평가지표 기준으로 최적화한다.
- 앙상블 후처리(가중 평균, rank averaging 등)로 최종 예측을 개선한다.
- 추론 시간 제약이 있는 경우 Quantization, Pruning, Knowledge Distillation을 검토한다.
- 공모전 제출 직전 체크리스트로 활용한다.

#### 7. 검증 전략
- 태스크 특성에 맞는 CV 폴드를 설계한다 (Stratified K-Fold, Group K-Fold, Time-series split 등).
- 로컬 검증 점수와 리더보드 점수의 괴리를 모니터링하고 원인을 분석한다.
- OOF(Out-of-Fold) 예측을 활용해 앙상블 가중치를 결정한다.
- 데이터 누수(Leakage) 가능성을 검토한다.
- 검증 전략 실수가 리더보드 overfitting의 주원인임을 인지하고 설계를 엄격히 관리한다.

#### 8. 오류 분석
- 틀린 예측 샘플을 직접 열람하여 실패 패턴을 분류한다.
- 오류가 특정 클래스·조건·구간에 집중되는지 확인한다.
- 원인을 데이터 부족, 레이블 오류, 피처 부재, 모델 표현력 부족 중 어디에 해당하는지 진단한다.
- 진단 결과를 근거로 다음 실험 방향(데이터 보강, 피처 추가, 아키텍처 변경 등)을 결정한다.
- 성능이 막혔을 때 수치 비교보다 우선적으로 수행한다.

### 에이전트 행동 원칙

- 항상 현재 베이스라인 성능을 먼저 파악한 뒤 개선 방향을 제안한다.
- 실험 1회마다 변경 변수를 최소화한다 (단일 변인 원칙).
- 제안은 "근거 → 변경 내용 → 예상 효과 → 검증 방법" 순서로 작성한다.
- 계산 비용 대비 기대 성능 향상을 항상 고려한다.
- 리더보드 또는 평가지표 기준에 맞는 최적화에 집중한다.

## 프로젝트별 파일 설계

### `projects/{project_name}/CLAUDE.md`

범용 에이전트를 상속하되, 아래 항목을 오버라이드·추가한다.

```markdown
# {프로젝트명} — Performance Agent 확장

## 프로젝트 개요
- 태스크 유형: (분류 / 회귀 / 생성 / 탐지 등)
- 평가지표: (예: F1-score, RMSE, mAP)
- 제출 형식 및 제약: (예: 추론 시간 제한, 모델 크기 제한)

## 현재 베이스라인
- 모델: ...
- 성능: ...
- 주요 미해결 문제: ...

## 프로젝트 특화 전략
- 이 태스크에서 유효한 것으로 확인된 기법: ...
- 이미 시도했고 효과 없었던 기법: ...
- 우선 탐색할 방향: ...

## 참조 자료
- context.md 참조
```

### `projects/{project_name}/context.md`

에이전트가 판단 근거로 사용하는 정적 컨텍스트.

```markdown
# {프로젝트명} — 컨텍스트

## 데이터셋
- 크기, 구조, 클래스 분포, 이상치 특성

## 평가 환경
- 플랫폼, 하드웨어 제약, 제출 횟수 제한

## 실험 이력
| 실험 ID | 변경 내용 | 성능 | 비고 |
|---------|----------|------|------|
| 001     | 베이스라인 | ... | ... |
```

## 워크플로우

```
새 프로젝트 시작
    │
    ▼
projects/{name}/ 폴더 생성
    │
    ├── CLAUDE.md 작성 (평가지표, 제약, 베이스라인 기록)
    └── context.md 작성 (데이터셋 정보, 초기 실험 이력 테이블)
    │
    ▼
에이전트 실행: 현재 코드·로그 분석 → 개선 방향 리스트 출력
    │
    ▼
단일 변인 실험 반복
    ├── 실험 결과 → context.md 이력 테이블에 추가
    └── 유효한 기법 → CLAUDE.md 특화 전략에 반영
    │
    ▼
최종 앙상블·제출 전략 수립
```

## 확장 계획

- `strategies/` 파일들은 실험을 거치며 유효성이 검증된 기법을 누적한다.
- 여러 프로젝트에서 반복 유효한 패턴은 범용 `CLAUDE.md`로 승격한다.
- 프로젝트 간 도메인이 다를 경우(CV / NLP / Tabular) 도메인별 서브 전략 파일을 추가할 수 있다.
