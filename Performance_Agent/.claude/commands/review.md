실험 제안을 실행 전에 검토한다. 문제가 있으면 차단하고 수정 방향을 제시한다.

## 검토 절차

### 1. context.md 읽기

`context.md`를 읽어 다음을 파악한다.
- 현재 베이스라인 성능
- 최근 3회 실험의 변경 내용과 결과
- 이미 시도해서 효과 없었던 기법 목록

### 2. 제안된 실험 파악

사용자가 명시하지 않았으면 "어떤 실험을 하려고 하는지 알려주세요"라고 묻는다.

### 3. 4가지 항목 검토

아래 각 항목에 ✅ 또는 ❌로 답한다.

**① 단일 변인 원칙**
- 이번 실험에서 변경하는 것이 정확히 하나인가?
- ❌이면: 어떤 변수를 분리해야 하는지 구체적으로 제시한다.

**② 중복 여부**
- 동일하거나 유사한 실험이 context.md 이력에 있는가?
- ❌이면: 어떤 실험과 겹치는지 실험 ID를 명시한다.

**③ 우선순위 적절성**
- 현재 상황에서 이 실험이 가장 기대 효과가 높은가?
- 성능 정체 중이라면 `/error-analysis`를 먼저 해야 하는가?
- ❌이면: 더 우선순위 높은 대안을 제시한다.

**④ 검증 전략 정합성**
- 이 실험의 결과를 올바르게 측정할 수 있는 검증 방법이 있는가?
- OOF 점수로 측정 가능한가? 데이터 누수 위험은 없는가?
- ❌이면: 수정해야 할 검증 설계를 제시한다.

### 4. 최종 판정

- 4개 모두 ✅ → "실험 진행 가능. `/new-experiment`로 시작하세요."
- 하나라도 ❌ → "실험 보류. [구체적 수정 사항]을 먼저 반영하세요."
